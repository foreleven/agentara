# 大纲设计检查清单

**用途**: 在完成阶段 2（大纲设计）后，用此清单自查大纲质量。

---

## 一、标题质量

- [ ] **标题符合公式**:
  - 实战教程: `[时间] + 从零做/搭建 + [技术] + 附源码/Demo`
  - 技术解析: `一文读懂 + [技术] + [亮点]`
  - 实战指南: `[技术] + 实战：[场景/问题]`

- [ ] **标题吸引眼球但不夸张**
- [ ] **标题明确技术和价值点**
- [ ] **标题符合 昕哥 的风格**（参考已发布文章）

---

## 二、结构合理性

### 整体结构

- [ ] **符合标准模板**:
  - Introduction → Demo → Overview → Technical Deep-dive → Advanced → Conclusion → CTA

- [ ] **章节数量适中**（通常 7-10 个大章节）
- [ ] **核心技术章节有 3-5 个**（不要太少或太多）

### 章节逻辑

- [ ] **章节之间有清晰的逻辑关系**（递进/并列/总分）
- [ ] **没有跳跃式讲解**（有铺垫和过渡）
- [ ] **从易到难，循序渐进**

### 章节命名

- [ ] **章节标题简洁明确**
- [ ] **使用中文为主，技术术语用英文**
- [ ] **避免模糊的标题**（如 "第三章"、"一些技术细节"）

---

## 三、内容覆盖度

### 调研结果的体现

- [ ] **覆盖了调研中的关键发现**（5-8 个 Key Findings）
- [ ] **融入了收集的代码示例**（至少 3-5 个）
- [ ] **体现了最佳实践和常见坑**

### 核心概念

- [ ] **Overview 中介绍了 3-5 个核心概念**
- [ ] **每个核心概念都有清晰的定义和说明**

### 实战内容

- [ ] **有 Demo 演示章节**（如适用）
- [ ] **有具体的代码示例规划**
- [ ] **有实际应用场景说明**

---

## 四、差异化

- [ ] **明确了与现有教程的不同之处**（至少 2-3 个差异点）
- [ ] **有独特的切入角度**（深度/实战/痛点/新特性）
- [ ] **避免了"又一篇 XXX 教程"的感觉**

---

## 五、篇幅规划

### 总字数

- [ ] **预估字数在合理范围内**:
  - 快速指南: 3000-5000
  - 标准教程: 5000-8000
  - 深度长文: 8000-12000

### 字数分配

- [ ] **Introduction 不要过长**（500-800 字）
- [ ] **核心技术章节字数均衡**（每个 1000-1500 字）
- [ ] **没有过短的章节**（少于 300 字）
- [ ] **没有过长的章节**（超过 2000 字，应考虑拆分）

---

## 六、代码示例规划

- [ ] **代码示例总数合理**（通常 5-10 个）
- [ ] **代码示例分布均匀**（不要集中在一个章节）
- [ ] **每个示例都有明确的功能说明**
- [ ] **标注了代码来源**（调研报告中的哪个示例）

**代码复杂度递进**:
- [ ] Overview: 简单示例（Hello World 级别）
- [ ] Technical Deep-dive: 核心功能示例（完整可运行）
- [ ] Advanced: 复杂场景示例（生产级）

---

## 七、图表规划

- [ ] **规划了必要的 Mermaid 图**（至少 2-3 个）
- [ ] **每个图表都有明确的用途**
- [ ] **图表类型多样**（流程图、架构图、序列图、表格）

**常见图表用途**:
- [ ] 概念关系 → Mermaid 流程图
- [ ] 架构设计 → Mermaid 架构图
- [ ] 数据流程 → Mermaid 序列图
- [ ] 对比分析 → Markdown 表格

---

## 八、每个章节的完整性

**对每个章节检查**:

- [ ] **有明确的章节目标**（这个章节要达成什么）
- [ ] **有核心内容列表**（3-5 个要点）
- [ ] **有写作要点提示**（具体的写作建议）
- [ ] **有预计字数**
- [ ] **有需要的素材说明**（代码示例/图表/链接）

---

## 九、风格一致性

### 符合 昕哥 的风格

- [ ] **读取了 `wikis/writing/technical-writing-guide.md`**
- [ ] **参考了已发布文章的结构**（`wikis/writing/published/`）
- [ ] **符合以下特点**:
  - 中文为主，技术术语用英文
  - 个人经历/痛点引入
  - 大量代码示例和 Mermaid 图
  - 系列文章链接和 CTA

### 目标读者明确

- [ ] **明确了目标读者**（初学者/进阶/专家）
- [ ] **语言难度匹配读者水平**
- [ ] **内容深度匹配读者需求**

---

## 十、实用性

### 可执行性

- [ ] **读者读完后能做什么**（明确的技能获取）
- [ ] **有完整可运行的 Demo**（如适用）
- [ ] **有 GitHub repo 链接**（如适用）

### 可操作性

- [ ] **每个技术点都有具体的代码示例**
- [ ] **有最佳实践和注意事项**
- [ ] **有常见坑和解决方案**

---

## 十一、外部链接

- [ ] **规划了官方资源链接**（文档、GitHub repo）
- [ ] **规划了参考文章链接**（社区博客）
- [ ] **规划了相关项目链接**（开源项目）
- [ ] **所有链接都来自调研报告**（有来源）

---

## 十二、系列文章关联

**如果是系列文章**:

- [ ] **明确了本文在系列中的位置**
- [ ] **有前序文章链接**（如适用）
- [ ] **有后续文章预告**（如适用）
- [ ] **内容与系列其他文章保持连贯**

---

## 十三、YAML Frontmatter

- [ ] **包含了必要的元数据**:
  - title（标题）
  - date（日期）
  - tags（标签，3-5 个）
  - series（系列名称，如适用）
  - demo_url（Demo 链接，如适用）
  - github_url（GitHub 链接，如适用）

---

## 不合格的典型表现

❌ **避免以下情况**:

1. **结构混乱**
   - 章节之间没有逻辑关系
   - 先讲高级内容，再讲基础概念

2. **重点不突出**
   - 核心技术点被淹没在大量细节中
   - 所有内容平铺直叙，没有重点

3. **缺少实战**
   - 只有概念介绍，没有代码示例
   - 只有理论，没有应用场景

4. **篇幅失衡**
   - Introduction 占了 2000 字，核心技术只有 1000 字
   - 某个章节过长（超过 2000 字），应拆分

5. **没有差异化**
   - 与现有教程没有明显区别
   - 只是重复官方文档的内容

6. **章节目标不明确**
   - 只有章节标题，没有说明要达成什么
   - 核心内容列表过于笼统

---

## 合格标准

✅ **大纲合格的标志**:

1. **结构清晰**: 符合标准模板，章节逻辑合理
2. **内容充实**: 覆盖调研的关键发现，有代码示例和图表
3. **有差异化**: 明确的独特价值，不是简单重复
4. **篇幅合理**: 预估字数在 5000-10000，分配均衡
5. **可执行**: 每个章节都有明确目标和写作要点
6. **符合风格**: 符合 昕哥 的写作风格和读者期待
