# 写作质量检查清单

**用途**: 在完成阶段 3（内容创作）后，用此清单自查文章质量。

---

## 一、完整性

### 章节完成度

- [ ] **所有大纲中的章节都完成了**
- [ ] **没有遗漏的章节**
- [ ] **没有空白或占位符内容**（如 "TODO", "待补充"）

### YAML Frontmatter

- [ ] **包含必要的元数据**:
  - title
  - date
  - tags（3-5 个）
  - series（如适用）
  - demo_url（如适用）
  - github_url（如适用）

---

## 二、内容质量

### Introduction（引言）

- [ ] **有个人经历/痛点引入**（建立共鸣）
- [ ] **说明了为什么要写这篇文章**
- [ ] **明确了读者能学到什么**
- [ ] **预告了文章结构**
- [ ] **字数在 500-800 之间**

### Demo 演示

- [ ] **有视觉化展示**（截图/GIF/视频）
- [ ] **简洁说明了核心功能**
- [ ] **提供了 Demo 链接或 GitHub repo**（如适用）
- [ ] **字数在 300-500 之间**

### Overview（概览）

- [ ] **有 TL;DR（一句话总结）**
- [ ] **介绍了 3-5 个核心概念**
- [ ] **每个概念都有清晰的定义**
- [ ] **有技术背景/历史说明**（简述）
- [ ] **有 Mermaid 概念图**（如适用）
- [ ] **字数在 800-1200 之间**

### 核心技术章节（3-5 个）

**对每个核心技术章节检查**:

- [ ] **开头有问题引入或目标说明**
- [ ] **使用小标题（###）拆分内容**
- [ ] **每个小节 300-500 字**
- [ ] **有 1-2 个代码示例**
- [ ] **代码前有说明，代码后有解释**
- [ ] **有 Mermaid 图表**（如适用）
- [ ] **有表格对比**（如适用）
- [ ] **字数在 1000-1500 之间**

### Advanced Topics（进阶话题）

- [ ] **有性能优化技巧**
- [ ] **有最佳实践总结**
- [ ] **有常见坑和解决方案**（表格形式）
- [ ] **有优化前后的代码对比**（如适用）
- [ ] **字数在 800-1200 之间**

### Conclusion（总结）

- [ ] **回顾了核心要点**（3-5 条）
- [ ] **有技术展望**（未来发展方向）
- [ ] **有系列文章链接**（如适用）
- [ ] **鼓励读者实践**
- [ ] **字数在 300-500 之间**

### Call-to-Action

- [ ] **有 GitHub Star 引导**（如适用）
- [ ] **有 ByteTech 关注引导**
- [ ] **有学习社区邀请**
- [ ] **有个人成就展示**（可选）
- [ ] **字数在 100-200 之间**

---

## 三、代码示例质量

### 代码数量和分布

- [ ] **代码示例总数合理**（5-10 个）
- [ ] **代码示例分布均匀**（不集中在一个章节）
- [ ] **符合复杂度递进**:
  - Overview: 简单示例
  - Technical Deep-dive: 核心功能示例
  - Advanced: 复杂场景示例

### 代码格式

- [ ] **使用正确的语言标识符**（```typescript, ```python）
- [ ] **代码缩进正确**
- [ ] **注释使用中文**
- [ ] **代码长度适中**（不要超过 50 行，如超过应拆分）

### 代码说明

- [ ] **代码前有说明**（这段代码做什么）
- [ ] **代码后有解释**（关键步骤说明）
- [ ] **标注了代码来源**（官方文档/开源项目/博客，可选但推荐）

### 代码可运行性

- [ ] **至少有 1 个完整可运行的示例**
- [ ] **提供了运行环境说明**（如 Node.js 版本、依赖包）
- [ ] **标注了需要修改的地方**（如 API Key、配置项）
- [ ] **没有语法错误**

---

## 四、图表质量

### Mermaid 图表

- [ ] **至少有 2-3 个 Mermaid 图**
- [ ] **每个图都有明确的用途**
- [ ] **图表类型多样**（流程图、架构图、序列图）
- [ ] **图表语法正确**（渲染正常）
- [ ] **节点命名清晰**（使用中文或有意义的英文）

### 表格

- [ ] **使用 Markdown 表格**
- [ ] **表格格式正确**（对齐、完整）
- [ ] **表头清晰**
- [ ] **内容简洁**（避免过长的单元格内容）

---

## 五、链接和引用

### 外部链接

- [ ] **第一次提到技术术语时加链接**
- [ ] **所有链接都有效**（没有 404）
- [ ] **链接使用 Markdown 格式**: `[文本](URL)`
- [ ] **官方文档链接优先**

### 内部链接

- [ ] **引用了自己的其他文章**（如适用）
- [ ] **系列文章之间有互相链接**
- [ ] **使用相对路径或完整 URL**

### 来源标注

- [ ] **关键发现有来源链接**（可追溯）
- [ ] **代码示例标注了来源**（可选但推荐）

---

## 六、语言和风格

### 语言规范

- [ ] **中文使用中文标点**：，。！？""（）《》
- [ ] **英文术语保持原样**：Agent, LangGraph, API
- [ ] **数字与单位之间有空格**：100 MB, 5 秒
- [ ] **中英文之间有空格**：使用 LangGraph 构建

### 写作风格

- [ ] **符合 昕哥 的风格**:
  - 第一人称（笔者、我们）
  - 口语化但专业（怎么样？让我们一起...）
  - 个人经历和实践经验
  - 大量代码示例和 Mermaid 图

### 一致性

- [ ] **术语使用一致**（不要一会儿 "状态图"，一会儿 "StateGraph"）
- [ ] **风格一致**（不要前面正式，后面随意）
- [ ] **结构一致**（每个章节的开头和结尾风格统一）

---

## 七、字数和篇幅

### 总字数

- [ ] **总字数在预期范围内**:
  - 快速指南: 3000-5000
  - 标准教程: 5000-8000
  - 深度长文: 8000-12000

### 章节字数

- [ ] **各章节字数符合预期**（参考大纲）
- [ ] **没有过短的章节**（少于 300 字）
- [ ] **没有过长的章节**（超过 2000 字，应考虑拆分）

### 篇幅平衡

- [ ] **Introduction 不要过长**（不超过 800 字）
- [ ] **核心技术章节字数均衡**
- [ ] **Technical Deep-dive 部分占总字数的 50-60%**

## 不合格的典型表现

❌ **避免以下情况**:

1. **只有概念，没有代码**
   - 全文都是文字描述，没有代码示例

2. **代码不可运行**
   - 代码有语法错误
   - 缺少关键步骤说明

3. **结构混乱**
   - 章节之间没有逻辑关系
   - 标题层级混乱

4. **风格不一致**
   - 前面正式，后面随意
   - 术语使用不一致

5. **缺少来源**
   - 关键发现没有来源链接
   - 代码示例不知道从哪来的

6. **篇幅失衡**
   - Introduction 占了 2000 字
   - 核心技术章节只有 500 字

7. **没有差异化**
   - 只是重复官方文档
   - 与现有教程没有区别

---

## 合格标准

✅ **文章合格的标志**:

1. **完整性**: 所有章节都完成，没有遗漏
2. **质量**: 代码可运行，图表清晰，链接有效
3. **风格**: 符合 昕哥 的风格，语言规范
4. **实用**: 读者能学到东西，能实践
5. **深度**: 不只是表面，有深入的技术讲解
6. **品牌**: 有个人特色，有社区互动
