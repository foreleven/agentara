# 阶段 0: 主题确认 📋

## 目标

明确写作主题、目标读者、核心价值，为后续调研和创作建立清晰的方向。

---

## 执行步骤

### 1. 与用户对话，收集信息

通过以下问题与用户互动（不需要一次性全问，根据用户的回答灵活调整）：

#### 关于主题

- **核心主题是什么？**
  - 例如：某个技术框架、实战教程、架构方法论、工具使用
- **有具体的版本或范围吗？**
  - 例如：LangGraph 1.0、Claude Code v2、Next.js 15
- **这个主题的新颖性在哪里？**
  - 是新发布的技术？还是对现有技术的新视角？
  - 有什么痛点或问题是这篇文章要解决的？

#### 关于读者

- **目标读者是谁？**
  - 初学者 / 有一定基础的开发者 / 架构师 / 特定领域专家
- **读者的痛点是什么？**
  - 缺少入门教程？现有教程不够深入？没有实战案例？
- **读者读完后应该能做什么？**
  - 搭建一个完整项目？理解核心概念？避免常见坑？

#### 关于文章类型

- **这是什么类型的文章？**
  - 实战教程（"30分钟从零做XXX"）
  - 技术解析（"一文读懂XXX"）
  - 最佳实践（"XXX 实战指南"）
  - 案例分享（"如何用XXX解决XXX问题"）
- **需要 Demo 吗？**
  - 如果需要，Demo 的复杂度如何？（简单示例 / 完整可运行项目）

#### 关于长度和深度

- **预期文章长度？**
  - 快速指南（3000-5000字）
  - 标准教程（5000-8000字）
  - 深度长文（8000-12000字）
- **技术深度如何？**
  - 概念介绍为主
  - 概念 + 实践结合
  - 深入源码或架构设计

#### 关于系列文章

- **这是独立文章还是系列的一部分？**
  - 如果是系列，之前有哪些文章？后续计划写什么？
- **是否需要与之前的文章保持连贯性？**

### 2. 阅读已发布文章作为参考

在征询用户意见的同时，可以阅读以下文件获取风格参考：

```
Read: wikis/writing/published/index.md
```

根据用户主题的类型，选择 1-2 篇相似的已发布文章快速浏览，了解：
- 标题风格
- 开头方式
- 结构模式
- 代码示例密度

读取模板：

### 4. 创建工作目录并保存

#### 生成文章 slug

根据主题生成 slug（小写、连字符分隔、去除特殊字符）：

- "LangGraph 1.0 实战教程" → `langgraph-1.0-tutorial`
- "从零搭建 Code Agent" → `build-code-agent-from-scratch`
- "Claude MCP 深度解析" → `claude-mcp-deep-dive`

#### 创建目录

```bash
mkdir -p wikis/writing/drafts/{article-slug}
```

#### 保存文件

```
Write: wikis/writing/drafts/{article-slug}/00-topic.md
```

---

### 5. 向用户确认

输出信息：

```markdown
✅ 主题确认完成！

**工作目录**: `wikis/writing/drafts/{article-slug}/`

**主题文件**: 已生成 `00-topic.md`

请查看 `00-topic.md`，确认以下内容：
- 文章主题和范围
- 目标读者
- 预期产出（Demo、长度、深度）

**确认无误后，我将进入阶段 1：深度调研。**
```

## 常见问题

### Q: 用户只说了一个简单的主题，没有详细需求怎么办？

A: 主动提问，引导用户思考：

- "这篇文章的目标读者是初学者还是有经验的开发者？"
- "你希望读者读完后能做什么？"
- "需要一个可运行的 Demo 吗？"

### Q: 用户说"随便写"或"你决定"怎么办？

A: 基于对用户风格的理解（参考 `memory/USER.md`）提出建议：

```markdown
根据你之前的文章风格，我建议：

**类型**: 实战教程（"30分钟从零做XXX"）
**长度**: 7000-9000字
**结构**: 概念介绍 → Demo 演示 → 核心技术讲解 → 进阶话题 → 总结
**产出**: 完整可运行的 GitHub repo

是否按这个方向进行？
```

### Q: 如果主题太宽泛（如"AI Agent"）怎么办？

A: 帮助用户缩小范围：

```markdown
"AI Agent" 是一个很大的话题，我们可以聚焦在：

1. **特定框架**: 如 "LangGraph 实现 Multi-Agent 系统"
2. **特定场景**: 如 "用 Agent 做代码生成"
3. **特定技术点**: 如 "Agent 的 Context 管理"

你更倾向于哪个方向？
```
