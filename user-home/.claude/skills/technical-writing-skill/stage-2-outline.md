# 阶段 2: 大纲设计 📝

## 目标

基于调研结果，设计清晰、合理的文章结构，明确每个章节的内容和目标。

### 3. 设计文章标题

**标题公式**（参考 `technical-writing-guide.md`）:

- **实战教程**: `[时间] + 从零做/搭建 + [技术] + 附源码/Demo`
  - 示例: "30分钟从零做个 LangGraph Agent 附源码"

- **技术解析**: `一文读懂 + [技术] + [亮点]`
  - 示例: "一文读懂 LangGraph 1.0 新特性"

- **实战指南**: `[技术] + 实战：[场景/问题]`
  - 示例: "LangGraph 实战：构建多 Agent 协作系统"

**标题要求**:
- 吸引眼球，但不夸张
- 明确技术和价值点
- 符合 昕哥 的风格（参考已发布文章）

---

### 4. 设计文章结构

#### 标准结构模板

根据文章类型选择结构：

##### 类型 1: 实战教程

```markdown
1. Introduction（引言）
   - 个人经历/痛点引入
   - 为什么要写这篇文章
   - 读者能学到什么

2. Demo 演示
   - 最终效果展示（截图/视频）
   - 核心功能说明
   - 技术栈

3. Overview（概览）
   - TL;DR（一句话总结）
   - 核心概念介绍
   - 技术背景/历史

4. Technical Deep-dive（技术深度）
   - 核心技术点 1
   - 核心技术点 2
   - 核心技术点 3
   - ...（3-5 个核心章节）

5. Advanced Topics（进阶话题）
   - 性能优化
   - 最佳实践
   - 常见坑和解决方案

6. Conclusion（总结）
   - 回顾核心要点
   - 未来展望
   - 系列文章链接

7. Call-to-Action
   - GitHub Star
   - ByteTech 关注
   - 学习社区邀请
```

##### 类型 2: 技术解析

```markdown
1. Introduction
2. 背景和问题
3. 核心概念详解
4. 实际应用案例
5. 与竞品对比
6. 最佳实践
7. Conclusion
```

#### 章节命名规范

**好的章节标题**:
- 使用中文为主，技术术语用英文
- 简洁明确，一目了然
- 符合递进逻辑

**示例**:
- ✅ "核心概念：StateGraph 与 MessageGraph"
- ✅ "实战：构建第一个 Agent"
- ✅ "进阶：多 Agent 协作机制"
- ❌ "第三章"（不明确）
- ❌ "一些技术细节"（太笼统）

### 6. 预估文章长度

根据大纲预估总字数：

| 文章类型 | 建议字数 |
|---------|---------|
| 快速指南 | 3000-5000 |
| 标准教程 | 5000-8000 |
| 深度长文 | 8000-12000 |

**计算方法**:
- Introduction: 500-800 字
- Demo: 300-500 字
- Overview: 800-1200 字
- 每个技术章节: 1000-1500 字
- Advanced: 800-1200 字
- Conclusion: 300-500 字

**调整策略**:
- 如果预估字数过多，考虑拆分成系列文章
- 如果预估字数过少，考虑增加实战案例或进阶内容

## 常见问题

### Q: 大纲设计有哪些常见错误？

A: 避免以下问题：

1. **结构混乱**:
   - 章节之间没有逻辑关系
   - 跳跃式讲解，缺少铺垫

2. **重点不突出**:
   - 核心技术点被淹没在大量细节中
   - 没有明确哪些是必须讲的，哪些是可选的

3. **缺少实战**:
   - 只有概念，没有代码示例
   - 只有理论，没有应用场景

4. **篇幅失衡**:
   - 某些章节过长，某些过短
   - Introduction 太长，技术深度不够

### Q: 用户说"大纲不够详细"怎么办？

A: 补充以下内容：

```markdown
收到！我会细化大纲。请问你希望我：

- [ ] 为每个章节增加更详细的要点？
- [ ] 明确每个章节的代码示例？
- [ ] 增加小节划分（如 3.1, 3.2）？
- [ ] 其他: __________
```

然后根据反馈更新 `02-outline.md`。

### Q: 大纲设计时如何体现差异化？

A: 基于调研结果，找出差异化角度：

- **角度 1: 深度**
  - 现有教程只讲表面，我们深入源码/架构

- **角度 2: 实战**
  - 现有教程只讲概念，我们做完整项目

- **角度 3: 痛点**
  - 现有教程没解决的坑，我们重点讲解

- **角度 4: 新特性**
  - 现有教程基于旧版本，我们聚焦最新版本

## 大纲设计技巧

### 技巧 1: 先总后分

- 先设计顶层结构（5-8 个大章节）
- 再细化每个章节的小节

### 技巧 2: 递进式讲解

- Introduction: 为什么（Why）
- Overview: 是什么（What）
- Technical Deep-dive: 怎么做（How）
- Advanced: 做得更好（Better）

### 技巧 3: 代码示例分布

- Overview: 简单示例（Hello World 级别）
- Technical Deep-dive: 核心功能示例（可运行的完整代码）
- Advanced: 复杂场景示例（生产级代码）

### 技巧 4: 图表规划

- 概念关系 → Mermaid 流程图
- 架构设计 → Mermaid 架构图
- 数据流程 → Mermaid 序列图
- 对比分析 → Markdown 表格
