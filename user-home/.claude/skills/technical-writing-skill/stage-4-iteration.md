# 阶段 4: 迭代优化 🔄

## 目标

根据用户反馈持续优化文章，直到用户满意。

---

## 执行步骤

### 1. 接收用户反馈

用户可能提出以下类型的反馈：

#### 类型 1: 内容修改

```markdown
用户: "第 3 章的代码示例太复杂了，能简化一下吗？"
用户: "缺少对 XXX 概念的解释"
用户: "这部分的表述不够清晰"
```

#### 类型 2: 结构调整

```markdown
用户: "第 4 章和第 5 章可以合并"
用户: "建议增加一个关于 XXX 的章节"
用户: "Advanced Topics 部分可以更详细"
```

#### 类型 3: 代码优化

```markdown
用户: "这段代码有 bug，需要修复"
用户: "希望增加更多代码注释"
用户: "能否增加一个完整的项目示例？"
```

#### 类型 4: 风格调整

```markdown
用户: "语气太正式了，希望更轻松一些"
用户: "希望增加更多个人经历"
用户: "Mermaid 图可以更清晰"
```

#### 类型 5: 细节完善

```markdown
用户: "缺少某个链接"
用户: "这个术语需要解释"
用户: "希望增加一个对比表格"
```

---

### 2. 分析反馈并制定修改计划

**步骤**:

1. **理解反馈的本质**
   - 用户想要什么？
   - 问题出在哪里？

2. **评估影响范围**
   - 是否只影响一个章节？
   - 是否需要调整整体结构？

3. **制定修改计划**
   ```markdown
   **修改计划**:

   1. 简化第 3 章的代码示例（预计修改 200 字）
   2. 在第 2 章增加 XXX 概念的解释（预计新增 300 字）
   3. 优化 Mermaid 流程图（第 4 章）
   4. 增加一个对比表格（第 5 章）

   **预计修改时间**: 20-30 分钟
   ```

4. **向用户确认计划**（可选，如修改较大）
   ```markdown
   收到！我计划做以下修改：
   1. XXX
   2. XXX
   3. XXX

   这样修改是否符合你的期待？
   ```

---

### 3. 执行修改

#### 读取当前草稿

```
Read: wikis/writing/drafts/{article-slug}/03-draft-v{N}.md
```

#### 使用 Edit 工具修改

**对于小范围修改**（几行到一个段落）:

```
Edit: wikis/writing/drafts/{article-slug}/03-draft-v{N}.md
old_string: {原文本}
new_string: {新文本}
```

**对于大范围修改**（整个章节或多处修改）:

读取整个文件，修改后写入新版本：

```
Write: wikis/writing/drafts/{article-slug}/03-draft-v{N+1}.md
```

---

### 4. 记录变更

#### 使用版本变更模板

读取模板：

```
Read: .claude/skills/technical-writing-skill/templates/version-changelog-template.md
```

填充模板并**追加**到草稿文件末尾：

```markdown
---

## Version Changelog

### v2 (2026-02-15)

**修改内容**:
1. 简化第 3 章代码示例（用户反馈：太复杂）
2. 增加第 2 章 XXX 概念解释（用户反馈：缺少说明）
3. 优化第 4 章 Mermaid 流程图

**字数变化**: 7500 → 7800 (+300)

**用户反馈**: "代码简化后更易理解了"

---

### v1 (2026-02-14)

**初版草稿**
```

---

### 5. 质量检查

**每次修改后都要检查**:

- [ ] 修改是否解决了用户的问题？
- [ ] 是否引入了新的问题？（如格式错误、链接失效）
- [ ] 代码修改后是否仍可运行？
- [ ] 字数是否在合理范围内？

**使用写作检查清单**（简化版）:

```
Read: .claude/skills/technical-writing-skill/checklists/writing-checklist.md
```

重点检查受影响的章节。

---

### 6. 保存新版本

**文件命名规范**:

- 第 1 版: `03-draft-v1.md`
- 第 2 版: `03-draft-v2.md`
- 第 3 版: `03-draft-v3.md`
- ...
- 最终版: `03-draft-final.md`

**为什么要保留多个版本**:
- 方便回溯（如果用户想恢复之前的版本）
- 追踪修改历史

---

### 7. 向用户展示修改结果

**输出格式**:

```markdown
✅ 文章修改完成（第 {N} 版草稿）！

**修改内容**:
1. {修改 1}
2. {修改 2}
3. {修改 3}

**字数变化**: {旧字数} → {新字数} ({+/-}变化)

**受影响的章节**:
- {章节 1}
- {章节 2}

**新版本文件**: `wikis/writing/drafts/{article-slug}/03-draft-v{N}.md`

**请查看修改后的版本，确认以下内容**:
- 修改是否符合预期？
- 是否还有其他需要调整的地方？

**如果满意，我可以生成最终版本并发布。**
```

---

### 8. 重复迭代

根据用户的新反馈，重复步骤 1-7，直到用户满意。

**迭代次数**:
- 通常 2-3 次迭代即可
- 如果超过 5 次，建议与用户沟通是否有根本性的方向问题

---

### 9. 生成最终版本

当用户表示满意后：

#### 创建最终版本

```
Read: wikis/writing/drafts/{article-slug}/03-draft-v{最新版本}.md
Write: wikis/writing/drafts/{article-slug}/03-draft-final.md
```

#### 发布到 published 目录

```
Read: wikis/writing/drafts/{article-slug}/03-draft-final.md
Write: wikis/writing/published/{文章标题}.md
```

**注意**:
- 文件名使用文章标题（URL 编码空格为 `%20`）
- 移除 Version Changelog 部分（或保留，根据用户偏好）

#### 更新索引

```
Read: wikis/writing/published/index.md
Edit: 在顶部添加新文章条目
```

```markdown
## 2026

### [新文章标题](./新文章标题.md)
**发布日期**: 2026-02-15

### [从 Chatbot 到 Long-horizon Agent](./从%20Chatbot%20到%20Long-horizon%20Agent.md)
**发布日期**: 2026-02-04
```

mv wikis/writing/drafts/{article-slug}/03-draft-v*.md wikis/writing/drafts/{article-slug}/archive/

建议保留以下文件作为记录：

- `02-outline.md`: 文章大纲
- `03-draft-final.md`: 最终草稿

## 常见问题

### Q: 用户提出了很多修改意见，如何优先处理？

A: 按优先级排序：

**优先级 1（必须修改）**:
- 技术错误（代码 bug、概念错误）
- 链接失效
- 格式问题

**优先级 2（重要）**:
- 结构调整
- 内容补充
- 代码优化

**优先级 3（锦上添花）**:
- 风格调整
- 细节完善

### Q: 如果修改导致字数大幅增加怎么办？

A: 两种策略：

1. **压缩其他部分**:
   - 删除冗余内容
   - 合并相似章节

2. **拆分文章**:
   - 如果字数超过 12,000，建议拆成系列
   - 与用户沟通拆分方案

### Q: 用户说"随便改改就行"怎么办？

A: 主动提出具体建议：

```markdown
我建议做以下优化：

1. 增加一个完整的项目示例（提升实用性）
2. 优化 Mermaid 图（提升可读性）
3. 补充最佳实践章节（提升深度）

是否按这个方向优化？
```

### Q: 如何判断文章已经"足够好"？

A: 参考以下标准：

- [ ] 用户表示满意
- [ ] 通过了写作检查清单
- [ ] 代码示例可运行
- [ ] 符合 昕哥 的风格
- [ ] 有差异化价值
- [ ] 字数在合理范围内（5000-10000）

## 迭代优化技巧

### 技巧 1: 小步快跑

- 每次只改一小部分
- 及时征求用户反馈
- 避免大改后用户不满意

### 技巧 2: 保持版本控制

- 每次修改都保存新版本
- 在文件末尾记录变更日志
- 方便回溯和对比

### 技巧 3: 主动提出改进

- 不要只是被动响应用户反馈
- 发现问题主动提出优化建议
- 展示专业性

### 技巧 4: 关注用户意图

- 理解用户反馈背后的真正需求
- 有时用户说的和想要的不完全一致
- 用专业判断提供更好的解决方案

## 发布后的工作

### 社交媒体发布（可选）

如果用户希望发布到社交媒体（如 ByteTech），可以帮助准备：

1. **提取文章摘要**（100-200 字）
2. **准备封面图**（如有）
3. **生成标签列表**（hashtags）

### GitHub 同步（可选）

如果用户有配套的 GitHub repo：

1. **同步代码**
2. **更新 README**
3. **发布 Release**（如适用）

### 数据跟踪（可选）

建议用户记录：
- 发布时间
- 阅读量
- 点赞/评论数
- GitHub Star 数（如有）

用于后续分析和优化。
