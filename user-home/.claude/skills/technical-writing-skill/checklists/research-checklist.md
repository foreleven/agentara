# 调研质量检查清单

**用途**: 在完成阶段 1（深度调研）后，用此清单自查调研质量。

---

## 一、资源覆盖度

### 官方资源

- [ ] **官方文档**: 阅读了至少 **3 章**核心内容（不是只看目录）
- [ ] **API 参考**: 查看了主要 API 的用法和参数说明
- [ ] **Changelog**: 了解了最新版本的新特性和变更
- [ ] **GitHub Repo**: 浏览了官方仓库的 README、examples/、docs/ 目录
- [ ] **官方博客**: 查看了官方发布的技术博客或公告（如有）

### 开源项目

- [ ] **分析了至少 2 个**相关开源项目
- [ ] **每个项目都记录了**:
  - 项目用途
  - Star 数（评估受欢迎程度）
  - 关键代码文件路径
  - 值得借鉴的设计或实现

### 社区内容

- [ ] **阅读了至少 3 篇**高质量技术博客/教程
- [ ] **查看了社区讨论**（如 GitHub Issues、Stack Overflow、Discord）
- [ ] **搜索了视频教程**（YouTube、B站等）
- [ ] **记录了每个来源的核心观点**

---

## 二、深度和广度

### 关键发现（Key Findings）

- [ ] **总结了 5-8 个**关键发现
- [ ] **每个发现都包含**:
  - 清晰的标题
  - 详细说明（不是一句话概括）
  - 重要性分析（为什么这个发现值得关注）
  - 至少 1 个来源链接

### 核心概念

- [ ] **列出了 3-5 个**核心概念
- [ ] **每个概念都有**:
  - 准确的定义
  - 官方文档链接
  - 与其他概念的关系（如适用）

### 最佳实践

- [ ] **总结了官方推荐的使用方式**（至少 2 条）
- [ ] **收集了社区经验**（至少 1 条）

### 常见坑

- [ ] **记录了至少 3 个**常见问题
- [ ] **每个问题都包含**:
  - 问题表现
  - 原因分析
  - 解决方案
  - 来源链接（如 GitHub Issue、Stack Overflow）

---

## 三、代码示例

- [ ] **收集了至少 3-5 个**可运行的代码示例
- [ ] **每个示例都标注了**:
  - 来源（官方文档 / 开源项目 / 博客）
  - 功能说明
  - 运行状态（可直接运行 / 需要修改）
- [ ] **至少有 1 个示例**是完整可运行的（已验证或有信心可以运行）

---

## 四、信息质量

### 来源可靠性

- [ ] **优先使用官方资源**（官方文档、GitHub repo）
- [ ] **社区资源来自知名平台**（Medium、Dev.to、知名博主）
- [ ] **避免过时内容**（尽量使用 2025-2026 年的资源）

### 信息准确性

- [ ] **每个关键发现都有来源链接**（可追溯）
- [ ] **技术名词使用准确**（与官方文档一致）
- [ ] **版本信息明确**（如 LangGraph 1.0，而非笼统的 "LangGraph"）

---

## 五、实用性

### 对文章创作的指导

- [ ] **基于调研，提出了文章结构建议**
- [ ] **明确了哪些内容是重点**（读者最关心的）
- [ ] **找到了差异化角度**（与现有教程的不同之处）
- [ ] **有了 Demo 设计思路**（如适用）

### 待深入的问题

- [ ] **记录了调研中发现但尚未完全理解的问题**
- [ ] **列出了可能的资源**（如需进一步调研）

## 不合格的典型表现

❌ **避免以下情况**:

1. **只看标题，没有深度阅读**
   - 只记录了 "看了某某文档"，但没有总结核心内容

2. **缺少来源链接**
   - 关键发现没有标注从哪里得出的结论

3. **代码示例不可运行**
   - 收集了代码，但没有验证是否可运行，也没有标注需要修改的地方

4. **信息过时**
   - 使用了 2024 年之前的资源，而技术已有重大更新

5. **调研范围过窄**
   - 只看了官方文档，没有查看社区内容和实际案例

---

## 合格标准

✅ **调研合格的标志**:

1. **资源多样**: 官方 + 开源 + 社区
2. **深度足够**: 不是浅尝辄止，而是深入理解核心概念
3. **来源清晰**: 每个关键发现都可追溯
4. **可执行性**: 至少有 1 个可运行的代码示例
5. **指导意义**: 调研结果能直接指导文章创作
