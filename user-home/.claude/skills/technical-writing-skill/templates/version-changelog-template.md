# Version Changelog

**用途**: 追踪文章的修改历史，每次迭代后追加一条记录。

---

## 如何使用

1. **第一次创建草稿时**，在文章末尾添加：

```markdown
---

## Version Changelog

### v1 ({YYYY-MM-DD})

**初版草稿**

**字数**: {数字}

**完成章节**:
- ✅ Introduction
- ✅ Demo 演示
- ✅ Overview
- ✅ 核心技术章节 1
- ✅ 核心技术章节 2
- ✅ 核心技术章节 3
- ✅ Advanced Topics
- ✅ Conclusion
- ✅ Call-to-Action

**代码示例**: {数量} 个
**Mermaid 图表**: {数量} 个
```

2. **每次迭代修改后**，在 Version Changelog 顶部添加新版本记录：

```markdown
### v{N} ({YYYY-MM-DD})

**修改内容**:
1. {修改 1}
2. {修改 2}
3. {修改 3}

**字数变化**: {旧字数} → {新字数} ({+/-}变化)

**受影响的章节**:
- {章节 1}
- {章节 2}

**用户反馈**: "{用户的原话}"

**修改原因**: {简要说明为什么这样改}
```

## 字段说明

| 字段 | 说明 | 必填 |
|------|------|------|
| **版本号** | v1, v2, v3... | ✅ |
| **日期** | YYYY-MM-DD 格式 | ✅ |
| **修改内容** | 列出所有修改点 | ✅ |
| **字数变化** | 旧字数 → 新字数 (+/-变化) | ✅ |
| **受影响的章节** | 哪些章节被修改了 | ✅ |
| **用户反馈** | 用户的原话（如有） | 可选 |
| **修改原因** | 为什么这样改 | 可选 |

---

## 注意事项

1. **版本号递增**: v1 → v2 → v3 → ... → final
2. **新版本在上**: 最新的版本记录在最上面
3. **保留历史**: 不要删除旧版本的记录
4. **简洁明确**: 每条修改说明要简洁但明确
5. **用户导向**: 记录用户反馈，体现以用户为中心

---

## 最终版本

当文章定稿后，可以选择：

**选项 1: 保留 Changelog**
- 展示迭代过程
- 体现专业性

**选项 2: 移除 Changelog**
- 发布时删除 Version Changelog 部分
- 保持文章简洁

根据用户偏好决定。
